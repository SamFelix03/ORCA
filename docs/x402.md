Service Provider Guide
Guide for service providers to accept payments from Kite Agent Passport agents via x402 or MPP.

Kite Agent Passport can pay any service that implements one of two open payment protocols:

x402 — HTTP 402-based payment protocol with on-chain settlement

MPP (Machine Payments Protocol) — an open standard by Stripe and Tempo for machine-to-machine payments (mpp.dev)

If your service already speaks either protocol, Passport agents can pay you today. If you are starting fresh, this guide walks through both options.

We encourage deploying on Kite chain for the fastest settlement and tightest integration with Passport.

How Passport Pays Your Service
An agent calls your service endpoint.

Your service returns 402 Payment Required with payment terms.

The Passport agent resolves payment using the user's approved spending session.

Your service receives proof of payment, verifies it, and delivers the response.

The specifics of step 2–4 depend on which protocol you implement.

Option 1: Accept Payments via x402
x402 uses HTTP 402 responses to negotiate on-chain payments between agents and services.

Payment Flow

Payment Flow
Your service returns a 402 Payment Required response with payment details.

The Passport agent obtains a signed payment authorization from the user's session.

The agent resends the request with the X-Payment header.

You verify the payment token and call a facilitator to settle on-chain.

The facilitator executes the transfer to your wallet.

You deliver the service response.

Implementing x402 in Your Service
Step 1: Return 402 Payment Required

When a request arrives without valid payment, return a 402 with your payment terms:


Copy
{
  "error": "X-PAYMENT header is required",
  "accepts": [{
    "scheme": "gokite-aa",
    "network": "kite-testnet",
    "maxAmountRequired": "1000000000000000000",
    "resource": "https://your-service.com/api/endpoint",
    "description": "Your API - Description of your service",
    "mimeType": "application/json",
    "outputSchema": {
      "input": {
        "discoverable": true,
        "method": "GET",
        "queryParams": {
          "param1": {"description": "Parameter description", "required": true, "type": "string"}
        },
        "type": "http"
      },
      "output": {
        "properties": {
          "result": {"description": "Result description", "type": "string"}
        },
        "required": ["result"],
        "type": "object"
      }
    },
    "payTo": "0xYourServiceWalletAddress",
    "maxTimeoutSeconds": 300,
    "asset": "0x38129cf4CE5E183eFF248F42A7D345Bb1B47621A",
    "extra": null,
    "merchantName": "Your Service Name"
  }],
  "x402Version": 1
}
Step 2: Receive and Verify the Payment Token

The agent resends the request with an X-Payment header containing a base64-encoded authorization:


Copy
curl -H "X-PAYMENT: eyJhdXRob3JpemF0aW9uIjp7..." \
  https://your-service.com/api/endpoint
Step 3: Settle via Facilitator


Copy
curl -X POST https://facilitator.pieverse.io/v2/settle \
  -H "Content-Type: application/json" \
  -d '{
    "authorization": {...},
    "signature": "0x...",
    "network": "kite-testnet"
  }'
Step 4: After confirming settlement, deliver your service response.

Option 2: Accept Payments via MPP
MPP (Machine Payments Protocol) is an open standard co-authored by Stripe and Tempo. It extends the HTTP 402 pattern with support for multiple payment methods, session-based billing, and IETF standardization.

Payment Flow
Agent requests a resource from your service.

Your service responds with a 402 challenge containing payment terms.

The agent submits a credential (proof of payment).

Your service verifies the credential, returns a receipt, and delivers the resource.

## ORCA `x402-provider` and Pieverse (live settlement)

Kite’s Step 3 `curl` to `https://facilitator.pieverse.io/v2/settle` with only `{ authorization, signature, network: "kite-testnet" }` is a **minimal illustration**. The Pieverse APIs used in production follow **x402 v2**: the JSON body includes top-level **`x402Version`**, **`paymentPayload`**, and **`paymentRequirements`** (see Coinbase/x402 facilitator OpenAPI), not the legacy one-liner alone.

For **Kite chain ID 2368**, Pieverse advertises support as **`exact`** + **`eip155:2368`** (query `GET https://facilitator.pieverse.io/v2/supported`). That CAIP-2 id matches `KITE_CHAIN_ID=2368` in ORCA agents.

The ORCA provider in `services/x402-provider/`:

- Returns HTTP **402** with `accepts[]` using **`scheme: exact`** and **`network: eip155:2368`** (override with `X402_CAIP_NETWORK` / `KITE_CHAIN_ID`).
- Decodes **`X-Payment`** (base64 JSON), supports **x402 v2** proofs and a **legacy** top-level `{ authorization, signature }` shape.
- Calls **`POST /v2/verify`** then **`POST /v2/settle`** on Pieverse with **`x402Version: 2`** in the body.

Required for non-stub mode: set **`X402_PAY_TO`**, **`PUBLIC_RESOURCE_URL`**, **`X402_PROVIDER_STUB=false`**, and fund Passport spending sessions. If **`kpass`** still emits only **`gokite-aa` / `kite-testnet`** proofs and settlement fails, Passport or a **`gokite-aa`**-aware facilitator may be required—enable **`X402_DEBUG_PAYMENT_SHAPE=true`** on the provider to inspect decoded header keys.

See also: [services/x402-provider/README.md](../services/x402-provider/README.md).
